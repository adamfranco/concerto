Importing Multiples

Collections - Importing Multiple Collections

Overview

The Collection Importer is capable of importing multiple Collections in a single import. This is possible only when the files are organized and compressed in the correct structure. Pay careful attention to the manner in which your archives are created, the importer will not be able to populate your data if it is not organized and labeled correctly.

Instructions

Using the same 'Import Collection(s)' link in Concerto as mentioned in 'Importing - Basics' you can import multiple Collections with the same form as long as your archive uses the proper structure:

Formatting

  • A single archive file containing one level of directories (named arbitrarily)
  • Each directory represents a separate Collection and should contain:
    • a single xml file named "metadata.xml"
    • individual data files needed for the import (images, audio, etc.)
  • Each "metadata.xml" file is formatted according to the correct Collection DOCTYPE definition located (here)

Admin - Importing All of Concerto

Overview

Importing an entire instance of Concerto can be a difficult task. There are a lot of things that can go wrong when importing such a large amount of data: limits set by various applications may cause the import to fail, other factors such as memory swapping and hard drive limitations may cause the import to grind to a halt. The option is given to attempt a single import of an entire instance of Concerto however this may not be the choice for you. There is another option, which is to use the 'Collection Importer' as mentioned above to import smaller groups of Collections at a time.

Importing Single Unarchived Files

Collections - Importing with Remote Data

Overview

Concerto allows the importing of data for which the data itself is stored remotely (i.e. files on remote servers). A single "metadata.xml" file can be imported on its own without compression or other files as long as all of the data it requires is referenced properly within the file. This can be done by writing the "metadata.xml" files by hand or programatically for the import. You can also organize the "metadata.xml" files into the appropriate structure to be imported as an archive, and thus import multiple Collections without the data in the archive. These are all options to facilitate the population of data into Concerto from other data storage or management environments.

Writing an XML file is not as hard as it sounds, and writing a programatic tool to create many XML files from other data management sources can be a rewarding task; although for the average person a quick (example) will suffice for their importing needs. There are DOCTYPE definitions (here), which are rules for the structure of the XML files that the importer can successfully handle. Using a tool like (XML Validator?) to validate your XML files against the DOCTYPE definitions will greatly reduce the stress involved with data population, and will help you understand the XML substructure. In the end there is only one way to guarantee that your XML file will work with the importer, and that is to import it.

  • Validate your XML files against our (DOCTYPE definitions)
  • Import your XML files

Importing non-XML Assets

Assets - Importing Exif Assets

Overview

The 'Exif Asset Importer' is available through the 'Import Assets' link on a Collection. Exif metadata is stored in image files by cameras and other digital imaging devices to describe many things about the image. These categories form a standard Schema for cataloging. Concerto's 'Exif Asset Importer' allows you to map the Exif metadata from it's Schema to a Schema of your choice, or even your own creation. Because Exif metadata is embedded in the images when importing the images only the Exif metadata and image will be imported.

Instructions

To import a group of Assets with their Exif metadata use the 'Import Assets' link on a Collection. Once at the form for importing Assets choose to import an archive, and choose Exif from the drop down menu of filetypes. As long as your archive follows the format presented below the images should be added to the Collection as individual Assets using the Schema defined by the "schema.xml" file described below.

Formatting

First the substructure of the archive itself. This is the easy part, take images with Exif metadata, and put them into a folder, add a "schema.xml" file for mapping Schemas, compress the archive, and you're done.

Now we arrive at the "schema.xml" part. The 'Exif Asset Importer' uses an XML file named "schema.xml" to map the Exif metadata Schema to the metadata Schema of your choice. This file has its own DOCTYPE definition (here) and an example (here). There is also a form in the works for facilitating the creation of this file, allowing you to map the Exif Schema to an existing Schema without writing the XML file yourself.

Assets - Importing Tab-Delimited Assets

Overview

The 'Tab-Delimited Asset Importer' is available through the 'Import Assets' link on a Collection. This importer provides the least complicated way to import new files into a Collection. As with the 'Exif Asset Importer' you need to have a metadata file, this time named "metadata.txt". This file is what tells the importer the Schema being used and what the metadata is for the Schema.

Instructions

To import a group of Assets listed in a Tab-Delimited "metadata.txt" file use the 'Import Assets' link on a Collection. Once at the form for importing Assets choose to import an archive, and choose Tab-Delim from the drop down menu of filetypes. As long as your archive follows the format presented below the entries should be added to the Collection as individual Assets using the Schema defined by the "metadata.txt" file described below.
